# NHP agent-registration golden vectors (temporary vendor)

`agent_registration_golden.json` is a **byte-identical temporary vendor** of the
qurl-conformance artifact `qurl-agent-registration-golden-vectors`
(`github.com/layervai/qurl-conformance`, `vectors/agent_registration_golden.json`).

It is copied verbatim (SHA-256-verified against the source) so
`crypto/golden.test.ts` can byte-fence the vendored NHP wire against the
cross-language reference **now**, without waiting on the published npm accessor.

## What is live now vs. at publish (tracked in #176)

**Live now — two fences keep the vendored copy honest:**

1. **Local-edit fence** — `crypto/golden.test.ts` pins this file's SHA-256
   (`FIXTURE_SHA256`), so a hand-edit that would quietly turn the cross-language
   byte fence back into a self-consistency check fails the unit suite.
2. **Cross-repo drift fence** — the `golden-drift` CI job
   (`scripts/check-golden-drift.mjs`) fetches the **canonical** upstream copy
   (`layervai/qurl-conformance` → `vectors/agent_registration_golden.json`) and
   fails if this vendored file has diverged from it. So the vendored copy cannot
   silently drift from upstream either.

Because the canonical file is not on the conformance default branch yet (it is in
`layervai/qurl-conformance#20`), the drift job resolves the PR branch first and
falls back to `main`, so it keeps working the moment that PR merges.

**At publish — a trivial re-point:** the published `@layervai/qurl-conformance`
package (0.1.2, already a devDependency and consumed by `portal.test.ts` via
`qv2Vectors()`) does **not yet** expose an `agentRegistrationVectors()` accessor.
Once it does, do the one-line swap documented at the top of
`crypto/golden.test.ts` (load via the accessor, the same pattern
`portal.test.ts` uses), then delete this fixture, its SHA pin, and the
`golden-drift` CI job + `scripts/check-golden-drift.mjs`.

Tracked in **layervai/qurl-typescript#176**: migrate the vendored fixture to the
published npm accessor (both fences already exist in-repo; #176 is now only the
accessor re-point).

Do not edit this file. It is source-of-truth data; any change must originate in
qurl-conformance and be re-vendored.
